# 0037 — Temporary artifacts have an explicit lifecycle

Status: accepted.

## Context

Vocabulary enrichment generates image and audio previews before the learner
decides whether to apply them. The API previously retained provider objects in
an unbounded job map and embedded Base64 bytes in every ready-job poll. That
mixed workflow state with large binary content, repeated copies on the JVM heap,
and gave neither bytes nor job metadata a hard capacity bound.

Spring Cache is already the replaceable acceleration boundary for exact query
results, while Attachments and Speech Assets are durable domain records. A
generated preview has different semantics from both: it is owner/session scoped,
short-lived, explicitly peeked or consumed, and safe to lose on restart.

## Decision

- Temporary generated bytes use the provider-neutral `TemporaryArtifactStore`
  contract, separate from `CacheManager`, semantic response caching, and the
  durable Attachment vault.
- Every artifact is scoped by owner, session, and category, carries immutable
  metadata plus defensively copied bytes, and supports metadata lookup,
  non-destructive peek, atomic destructive consume, individual deletion, and
  session cleanup.
- `apps/api` wires a conditional Caffeine provider. It expires after write,
  records statistics, rejects content above 16 MiB, and uses weighted eviction
  to bound both actual bytes at 64 MiB and effective entry count at 100. The
  vocabulary job registry separately caps active jobs at 100.
- Enrichment job JSON carries typed artifact references, never Base64 content.
  A job-scoped authenticated endpoint serves only referenced bytes with
  `no-store`, `nosniff`, and a sandboxing content policy.
- Preview Apply reads without consuming and deletes the job session only after
  the database transaction commits. Rollback keeps the preview retryable.
  Discard, failure, and expiry remove every artifact in the session, and a late
  provider result cannot publish after its job is no longer active.
- API restart remains an accepted loss boundary and never persists un-applied
  previews.

## Consequences

Polling cost no longer scales with image/audio size and the API has an explicit
heap budget for generated content. Consumers depend on a lifecycle contract
rather than Caffeine, so a later multi-replica implementation can place private
bytes in S3 while keeping owner, expiry, and atomic claim metadata in PostgreSQL
or Redis.

Durable Attachment storage is intentionally not generalized here. Its existing
`DATABASE`, reserved `LOCAL`, and reserved `S3` choices require a separate
content-store increment with transactional compensation and streaming; S3 is
not treated as an interchangeable in-memory cache.
