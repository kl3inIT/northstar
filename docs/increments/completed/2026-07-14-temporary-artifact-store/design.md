# Temporary Artifact Store Design

Status: completed on 2026-07-14.

Date: 2026-07-14

## Outcome

Northstar gets a provider-neutral lifecycle for short-lived generated bytes.
Vocabulary enrichment becomes the first consumer: job polling returns small
artifact references instead of repeatedly embedding Base64 image and audio data.
Nothing is persisted until the learner explicitly applies the preview.

This increment does not change the durable Attachment vault, speech assets,
learner recordings, exact application caches, or semantic response cache.

## Research

- Northstar currently retains enrichment image and audio objects inside
  `VocabEnrichmentJobService` and Base64-encodes them on every ready-job view.
- The production API container is limited to 1,200 MiB and the JVM uses at most
  75 percent of container RAM. The image adapter accepts up to 12 MiB, so an
  unbounded job map can create material heap pressure.
- Caffeine 3.2.4 is the latest Maven Central release as of this design and is
  already resolved by the build. Its source confirms that `maximumWeight` plus
  a `Weigher`, `expireAfterWrite(Duration)`, statistics, a thread-safe
  `Cache.asMap()`, and atomic map removal are supported. It also forbids using
  `maximumSize` and `maximumWeight` together.
- The AgentCore artifact-store pattern was inspected at commit
  `7593eb2ce6fbd3585271379e84ecd8059abc792e`. Northstar adopts lifecycle ideas,
  not its AWS runtime contract or implementation: session/category isolation,
  peek versus consume, TTL, immutable content, and a replaceable provider.

## Settled decisions

### D1 - Temporary artifacts are a distinct boundary

`TemporaryArtifactStore` is not Spring `CacheManager`, semantic caching, or
durable file storage. Temporary artifacts are disposable generated outputs;
attachments and applied speech remain authoritative PostgreSQL-backed records.

### D2 - Core owns the port; apps/api wires the default provider

The provider-neutral contract and immutable value types live in the open
`core.artifact` Modulith module. `apps/api` owns the Caffeine adapter and its
configuration because vocabulary preview jobs execute only in the API process.
The default bean is conditional on a missing `TemporaryArtifactStore`, so a
future multi-replica provider can replace it without changing consumers.

### D3 - Every artifact is owner, session, and category scoped

An `ArtifactScope` contains a non-secret owner scope, session id, and category.
The current single-user API uses one stable owner scope and the enrichment job
id as the session. Callers must present the exact scope and artifact id to read,
consume, or delete content. Session deletion removes every category belonging
to that owner and session.

### D4 - Stored content is immutable and safe to serve

Writes and reads defensively copy byte arrays. Metadata includes a generated id,
safe filename, normalized media type, byte size, SHA-256, creation time, and
expiry. Empty data, unsafe filenames, header-breaking media types, and
oversized content are rejected before insertion. The vocabulary endpoint serves
only an artifact already referenced by that job, with `no-store`, `nosniff`, a
sandboxing CSP, an inline disposition, and the stored media type.

### D5 - Memory is bounded by bytes and effective entry count

Defaults are a 30-minute write TTL, 16 MiB per artifact, 64 MiB total weighted
capacity, and 100 entries. Because Caffeine forbids combining maximum size and
maximum weight, each entry's weight is the greater of its byte size and one
hundredth of total capacity. This enforces both the real byte ceiling and an
effective entry ceiling under concurrent writes. Caffeine statistics are
recorded. Configuration values are validated at startup. The separate job
metadata registry has a 100-job semaphore bound so pending provider work cannot
grow without limit before it produces bytes.

### D6 - Job polling never transports artifact bytes

`VocabEnrichmentJobView` returns typed image, word-audio, and example-audio
references containing id, URL, media type, and size. The SPA uses those URLs in
`img` and `audio` elements. Polling remains small regardless of generated file
size; the binary endpoint is fetched only when the browser renders or plays it.

### D7 - Apply, discard, failure, and cancellation clean up deliberately

Generation publishes an artifact only while its job is still active. Discarded
or expired jobs cannot publish late provider results. Failures delete any
partially produced session artifacts. Apply peeks content, persists it through
the existing Attachment/Speech services, and deletes the job session only after
the surrounding transaction commits. A failed transaction therefore leaves the
preview available for retry. Discard removes both job metadata and artifacts.

### D8 - S3-compatible does not mean pretending S3 is Caffeine

The port avoids Caffeine types and uses immutable objects, explicit scope,
metadata, TTL, and destructive consume semantics that a remote implementation
can reproduce. Actual S3 support is deferred. A multi-instance provider will
need authoritative metadata/claim state in PostgreSQL or Redis plus private
object bytes in S3; S3 alone cannot provide owner validation and atomic consume.
Durable Attachment `DATABASE`/`S3` storage remains a separate future increment.

## Failure behavior

- A full store evicts older/colder previews rather than exceeding the heap
  budget. A missing referenced artifact is reported as an expired preview and
  cannot be applied.
- Provider failures return the existing failed-job state without exposing raw
  provider responses or secrets.
- Artifact download for an unknown job, an unreferenced id, or an expired value
  is `404`; it never searches by id outside the job scope.
- API restart loses pending previews by design and persists nothing.

## Verification

1. Contract validation, defensive-copy, scope isolation, TTL, weighted bounds,
   session deletion, and exactly-once concurrent consume tests.
2. Enrichment tests for transient generation, reference-only views, binary
   serving, apply-after-commit cleanup, discard, failure, and late publication.
3. Web unit/type checks proving the preview uses reference URLs and no Base64.
4. Spring Modulith verification and API context-load tests.
5. Manual browser enrichment preview covering image/audio render and Apply or
   Discard when provider credentials are available; otherwise deterministic
   controller/service tests remain the non-live gate.
6. Consolidate the Study spec, test matrix, a new decision, roadmap, and move
   this increment to `completed/` before publishing.
