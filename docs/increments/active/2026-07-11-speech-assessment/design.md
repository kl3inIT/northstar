# Speech Assessment (Pronunciation + Speaking Practice) — Design

Owner context: this design consolidates decisions made in a working session
on 2026-07-11 (research + user agreement). Everything an implementer needs is
in this file, the linked specs/decisions, and the plan beside it. Read first:
[docs/specs/domains/study.md](../../../specs/domains/study.md),
[decision 0015](../../../decisions/0015-vocab-memory-uses-recall-probability-not-due-dates.md),
[decision 0016](../../../decisions/0016-writing-grader-uses-sourced-anchored-comparative-grading.md).

## Goal

Two user-facing capabilities on `/study`, sharing one Azure Speech
integration:

1. **Pronunciation per vocab card** — the user records themselves reading a
   card's front; the app returns per-word and per-phoneme accuracy scores
   (works for Chinese incl. tones and English).
2. **Speaking practice** — an examiner-style question, the user records a
   spoken answer (≤60s), the app returns Azure delivery scores
   (pronunciation/fluency/prosody) plus LLM content feedback grounded in the
   user's existing error corpus; the attempt is persisted and the spoken
   grammar errors join the SAME corpus the grammar drills use.

User profile that shaped this: single user, IELTS band 4–4.5 + HSK4, no
daily rituals, review-through-chat philosophy, VPS deploy, all provider keys
server-side.

## Decisions (with rationale — do not re-litigate, supersede via a new decision doc)

### D1 — Azure Speech Pronunciation Assessment, not LLM-from-transcript, not OSS GOP

- LLM-on-transcript CANNOT judge pronunciation/fluency: the transcriber
  auto-corrects the learner's speech, destroying the signal. This is why the
  study increment deferred speaking (see spec).
- Azure Pronunciation Assessment is purpose-built: per-phoneme / syllable /
  word / full-text accuracy, fluency (measured from pauses), prosody (en-US),
  and an **unscripted mode** (no reference text) that adds content scores —
  designed for spoken-exam scenarios.
  Docs: https://learn.microsoft.com/en-us/azure/ai-services/speech-service/how-to-pronunciation-assessment
- zh-CN is supported (phoneme names via SAPI) — required for HSK.
- Open-source alternatives (Kaldi `gop_speechocean762`, GOPT transformer,
  wav2vec2 CAPT) were evaluated and rejected for V1: English-centric (no
  credible Mandarin tone scoring), require a self-hosted acoustic-model
  sidecar, quality below Azure — disproportionate for a single user.
- Cost: F0 free tier = 5 audio hours/month, billed per second. Card reads
  are 2–4s, speaking answers ≤60s: a heavy month stays inside free tier.

### D2 — No IELTS band claims from speech scores (hard rule)

Azure returns 0–100 scores. There is NO validated mapping from those to
IELTS 0–9 bands. Present per-dimension scores labeled unofficial; never
synthesize a "speaking band". This extends decision 0016's range philosophy
(even GPT-4 at kappa 0.811 with official examiners has 29% of essays >0.5
band off — inventing precision misleads the learner).

### D3 — Speaking is PRACTICE with measured delivery, not an exam simulation

- The full conversational examiner experience (realtime, interruptions,
  audio-native) is better in ChatGPT Advanced Voice — do not rebuild it.
  Northstar's edge is measurement + memory: numbers per attempt, persistence,
  the error corpus, review integration.
- V1 shape: a Speaking practice surface on `/study` (NOT inside chat — chat
  has no audio-output/audio-attachment plumbing and adding it is a separate
  increment). One question at a time → record → assess → feedback → history.
- Question source: generated by the existing ChatClient (an endpoint returns
  one Part 1/2/3-style question, optionally themed by the user's disciplines)
  — never a static bank to maintain.

### D4 — Spoken errors join the one error corpus

`grammar_weaknesses` currently aggregates `writing_feedback.topErrors`. A new
`speaking_feedback` table mirrors the topErrors JSON convention, and the
aggregation UNIONS both sources. Result: grammar drills train the user's
combined written+spoken error profile with zero new drill machinery.

### D5 — Provider isolation follows the web-research precedent (decision 0012)

- `core.study` owns a provider-neutral contract (`SpeechAssessor` interface +
  result records). Core never imports Azure types.
- The Azure adapter lives in a new Gradle module `integrations/speech-azure`
  (mirror `integrations/web-openai`'s build setup), wired only by `apps/api`.
  mcp/worker never see it.
- Key/region via env only: `AZURE_SPEECH_KEY`, `AZURE_SPEECH_REGION`
  (already present in the developer's `.env`; region is `eastasia` — the key
  is region-bound, a mismatch returns 401). Add both to `.env.example` and
  `docker/env.server.example` with placeholder values. Never expose to the
  browser; never log.
- Feature-flag behavior: when the key is blank, the api boots fine and the
  speech endpoints return 503 with a clear "not configured" ProblemDetail
  (ObjectProvider pattern — see how `search` degrades without an embedding
  model).

### D6 — Audio format: WAV PCM 16 kHz mono, produced client-side

Azure's REST/SDK accepts WAV/PCM natively; compressed formats need GStreamer
server-side (rejected). The web app already captures raw PCM via an
AudioWorklet for live dictation (24 kHz) — reuse that capture path, resample
to 16 kHz mono, wrap in a WAV header in the browser, upload
`audio/wav` ≤ ~2 MB (60s of 16k mono PCM ≈ 1.9 MB). Enforce a 60s cap in the
recorder UI and reject >75s server-side.

### D7 — Azure transport: REST, not the Java Speech SDK

Use the short-audio REST endpoint with the pronunciation-assessment header
(the SDK pulls in a native library and a session model we don't need for
one-shot files):

- URL: `https://{region}.stt.speech.microsoft.com/speech/recognition/conversation/cognitiveservices/v1?language={locale}&format=detailed`
- Headers: `Ocp-Apim-Subscription-Key`, `Content-Type: audio/wav; codecs=audio/pcm; samplerate=16000`,
  and `Pronunciation-Assessment: <base64 of JSON params>`.
- Scripted (card) params: `{"ReferenceText":"<front>","GradingSystem":"HundredMark","Granularity":"Phoneme","Dimension":"Comprehensive"}`.
- Unscripted (speaking) params: same minus ReferenceText (empty string),
  plus `"EnableContentAssessmentWithTopic":true,"Topic":"<question>"` for
  content scores, and `"EnableProsodyAssessment":true` (en-US only).
- Locale: `zh-CN` when the card front contains CJK characters, else `en-US`
  (helper: any codepoint in CJK Unified Ideographs). Speaking is always
  `en-US` in V1 (IELTS).
- IMPORTANT for the implementer: verify the exact param casing and response
  JSON shape against the official docs page above BEFORE coding the parser —
  do not trust this sketch for field names. The response carries
  `NBest[0].PronunciationAssessment` (Accuracy/Fluency/Prosody/Completeness)
  plus `Words[] → Phonemes[]` with per-unit `PronunciationAssessment.AccuracyScore`,
  and for unscripted, `ContentAssessment` (Vocabulary/Grammar/Topic scores).
  The 2026-07-11 session verified the key works via
  `POST https://eastasia.api.cognitive.microsoft.com/sts/v1.0/issueToken` → 200.

### D8 — What is persisted

- Card pronunciation attempts: NOT persisted in V1 (live feedback only) —
  avoids a new table for a feature whose value is the immediate signal. The
  user can re-run anytime. (V1.5 may add per-card pronunciation history.)
- Speaking attempts: persisted in `speaking_feedback` (V32) — question,
  transcript (from Azure's recognition), the score set, topErrors JSON,
  LLM feedback summary, grader model id (pinned, same property pattern as
  writing), audio NOT stored (the receipt-image precedent: media in,
  structure out).
- Every speaking attempt also auto-logs a study session
  (skill=Speaking, kind=PRACTICE, durationMinutes from audio length rounded
  up, source=ASSISTANT) so the Log tab and weekly review see it.

## Entities (migration V32 — check current max migration first; use next free number)

```sql
CREATE TABLE speaking_feedback (
    id             UUID             PRIMARY KEY,
    submitted_at   TIMESTAMPTZ      NOT NULL,
    question       VARCHAR(1000)    NOT NULL,
    transcript     VARCHAR(8000)    NOT NULL,
    -- Azure delivery scores, 0-100, NULL when Azure omitted one
    pronunciation  DOUBLE PRECISION,
    fluency        DOUBLE PRECISION,
    prosody        DOUBLE PRECISION,
    -- Azure content scores 0-100 (vocabulary/grammar/topic), JSON text
    content_scores VARCHAR(1000)    NOT NULL,
    top_errors     VARCHAR(4000)    NOT NULL,
    summary        VARCHAR(4000)    NOT NULL,
    grader_model   VARCHAR(64)      NOT NULL,
    created_at     TIMESTAMPTZ      NOT NULL,
    updated_at     TIMESTAMPTZ      NOT NULL,
    version        BIGINT           NOT NULL DEFAULT 0
);
CREATE INDEX speaking_feedback_submitted_idx ON speaking_feedback (submitted_at);
```

topErrors JSON shape must MATCH writing_feedback.topErrors exactly
(`[{"label","quote","fix"}]`) so `WritingService.grammarWeaknesses()` can
union both tables without a parser change (quote = verbatim fragment of the
TRANSCRIPT).

## Core contracts (`core.study`)

```java
/** Provider-neutral speech assessment; Azure adapter lives in integrations. */
public interface SpeechAssessor {
    /** Scripted: how well the audio matches referenceText. */
    PronunciationResult assessReading(byte[] wavAudio, String referenceText, String locale);
    /** Unscripted: free speech about a topic. */
    SpokenAnswerResult assessSpokenAnswer(byte[] wavAudio, String topic);
}

record PronunciationResult(double accuracy, double fluency, Double prosody,
        String recognizedText, List<WordScore> words) { }
record WordScore(String word, double accuracy, String errorType,
        List<PhonemeScore> phonemes) { }
record PhonemeScore(String phoneme, double accuracy) { }
record SpokenAnswerResult(String transcript, double pronunciation, double fluency,
        Double prosody, Double vocabulary, Double grammar, Double topic,
        List<WordScore> words) { }
```

`SpeakingService` (core.study, @Service): persistence + corpus reads, mirrors
`WritingService`. `SpeakingCoach` (core.study, NOT a component — the
CaptureService/WritingGrader precedent): orchestrates assess → LLM content
feedback → persist → auto-log session; wired by `apps/api` with ChatClient +
SpeechAssessor + grader model.

LLM content feedback prompt: reuse the calibration rules idea from the
writing rubric (never invent weaknesses; quote the transcript verbatim in
topErrors; spoken register is fine — do NOT flag contractions or informal
tone as errors; ~3 sentence summary; explicitly worded unofficial). Inject
the existing prior-error corpus (writing + speaking) the same way
WritingGrader does. Validate structurally in code (quotes verbatim in
transcript, whitespace-tolerant — reuse the WritingGrader normalize pattern);
run the existing `WritingFaithfulnessEvaluator` shape against the transcript
(one corrective retry, then loud failure) — factor the evaluator-optimizer
helper if sharing is cleaner than copying.

## API (apps/api, StudyController or a new SpeechController under /api/study)

- `POST /api/study/vocab/{id}/pronunciation` — multipart `audio` (wav) →
  `PronunciationResult` with referenceText = card front, locale by CJK
  detection. 404 unknown card, 400 bad/oversized audio, 503 not configured.
- `POST /api/study/speaking/question` — body `{ "part": 1|2|3 }` → one
  generated question `{ question }` (ChatClient; theme may draw from
  discipline names; keep the prompt short and pinned in code).
- `POST /api/study/speaking/attempts` — multipart `audio` + form field
  `question` → full result (Azure scores + feedback + persisted row id).
- `GET /api/study/speaking` / `DELETE /api/study/speaking/{id}` — history,
  same shape philosophy as `/api/study/writing`.
- Regenerate contract + `pnpm -C web gen:api` (boot with
  `NORTHSTAR_AUTH_ENABLED=false`, curl `/v3/api-docs` — mind the zombie-JVM
  port-8888 trap in docs/guidelines).

## Web (`web/src/features/study/`)

- **Vocabulary tab**: mic icon per card row → dialog: card front + reading
  shown, record (waveform or simple timer), stop, submit → per-word chips
  colored by accuracy (≥80 green, 60–79 amber, <60 red), expandable phoneme
  rows, overall accuracy/fluency. "Unofficial · Azure Speech" caption.
- **New Speaking tab** (4th tab, PenLine→Mic icon): stats strip (attempts,
  latest pronunciation/fluency, trend), "Get a question" (part selector),
  recorder (≤60s, live countdown), result card (delivery scores + content
  scores + feedback + transcript with error quotes highlighted), history
  table with detail dialog + delete — mirror the Writing tab patterns.
- Recorder: reuse/adapt the AudioWorklet PCM capture used by live dictation;
  downsample to 16 kHz mono, WAV-encode in the browser (small hand-rolled
  encoder — no new dependency), enforce the 60s cap.
- Assistant system prompt: add one routing line — speaking practice requests
  ("luyện nói") → point the user at the Speaking tab (chat cannot take audio
  in V1); after practicing, spoken errors automatically join grammar drills.

## Out of scope (deliberate)

- TTS/dictation (OpenAI `gpt-4o-mini-tts`) — separate increment.
- Audio in chat (either direction), realtime conversation — ChatGPT voice
  covers conversational practice; revisit only if measurement-in-conversation
  becomes necessary.
- Storing audio, per-card pronunciation history, HSK bulk import (separate
  backlog item), any IELTS band estimate for speaking (see D2 — hard rule).
